Threat Modelling with STRIDE: A Developer's Hands-On Walkthrough

Threat modelling feels abstract until you run it on a real system. Walk through a complete STRIDE analysis of a web API — spoofing, tampering, repudiation, information disclosure, DoS, and elevation of privilege.

AR

Alex Rivera

Jan 15, 2024 12 min read

Most security vulnerabilities aren't discovered by penetration testers — they're discovered by attackers. Threat modelling flips the timeline: instead of finding vulnerabilities after they're deployed, you systematically identify them during design. STRIDE is the most widely used framework for this, and it's approachable enough for any developer to run in a sprint planning session.

The System We're Modelling

We'll threat model a simple payment API. The system has four components:

  • Client — a mobile app that calls the API
  • API Gateway — handles authentication and rate limiting
  • Payment Service — processes transactions
  • Database — stores transaction records

Data flows: Client → API Gateway → Payment Service → Database

The goal of STRIDE is not to find every possible attack. It's to systematically apply six threat categories to every component and data flow, so you don't miss entire threat classes.

S — Spoofing

Spoofing means impersonating a user, system, or process. For our payment API:

  • Can an attacker claim to be a legitimate user? (Weak auth, stolen tokens)
  • Can a malicious service impersonate the Payment Service to the database?
  • Can the client be tricked into talking to a fake API Gateway?

Mitigations: Strong authentication (MFA, short-lived JWTs), mutual TLS for service-to-service, certificate pinning in the mobile client, HTTPS everywhere.

# Mitigation: Short-lived JWT with refresh token rotation
import jwt
from datetime import datetime, timedelta

def create_access_token(user_id: str) -> str:
    payload = {
        'sub': user_id,
        'iat': datetime.utcnow(),
        'exp': datetime.utcnow() + timedelta(minutes=15),  # short expiry
        'type': 'access'
    }
    return jwt.encode(payload, SECRET_KEY, algorithm='HS256')

T — Tampering

Tampering means modifying data in transit or at rest. For our system:

  • Can an attacker modify the transaction amount in flight?
  • Can a rogue process alter transaction records in the database?
  • Can the payment request be replayed with a modified payload?

Mitigations: TLS for all transport, database access controls (least privilege), request signing with HMAC to prevent payload modification, idempotency keys to prevent replay attacks.

# Mitigation: HMAC request signing
import hmac
import hashlib

def sign_request(payload: str, secret: str, timestamp: str) -> str:
    message = f"{timestamp}.{payload}"
    return hmac.new(secret.encode(), message.encode(), hashlib.sha256).hexdigest()

def verify_request(payload, signature, secret, timestamp, tolerance_seconds=300):
    expected = sign_request(payload, secret, timestamp)
    age = abs(int(time.time()) - int(timestamp))
    if age > tolerance_seconds:
        raise ValueError("Request too old — possible replay attack")
    return hmac.compare_digest(expected, signature)

R — Repudiation

Repudiation is the ability to deny performing an action. In a payment system this is critical — users must not be able to deny transactions they made, and the system must be able to prove what happened.

  • Can a user deny placing an order that was charged?
  • Can a developer deny making an unauthorised configuration change?

Mitigations: Immutable audit logs with cryptographic integrity (append-only log storage, log signing), timestamped records with actor identity, retained request/response logs for disputed transactions.

I — Information Disclosure

Information disclosure means exposing data to unauthorised parties. Key questions:

  • Do error responses leak stack traces, database schema, or internal IPs?
  • Are payment card numbers stored or logged anywhere?
  • Can user A access user B's transaction history?
  • Are logs shipped to a third-party service that could be breached?

Mitigations: Generic error messages to clients, structured logging that never captures PII/PCI data, row-level security in the database, RBAC on all data access endpoints.

# Bad — exposes internal details
@app.errorhandler(Exception)
def handle_error(e):
    return jsonify({'error': str(e), 'traceback': traceback.format_exc()}), 500

# Good — generic message, internal logging
@app.errorhandler(Exception)
def handle_error(e):
    error_id = str(uuid.uuid4())
    logger.exception(f"Unhandled error {error_id}")
    return jsonify({'error': 'An internal error occurred', 'id': error_id}), 500

D — Denial of Service

DoS attacks prevent legitimate users from accessing the service. For a payment API:

  • Can an attacker exhaust database connections with expensive queries?
  • Can a single user submit thousands of requests per second?
  • Can a malformed request cause the service to crash?

Mitigations: Rate limiting per user and per IP at the gateway, query timeouts and resource limits at the database, input validation that rejects oversized payloads early, circuit breakers between services.

E — Elevation of Privilege

Elevation of privilege means gaining capabilities beyond what was granted. This is the most dangerous STRIDE category:

  • Can a regular user call admin-only endpoints?
  • Can an attacker craft a JWT with elevated roles?
  • Can the Payment Service access tables it shouldn't have access to?

Mitigations: Authorisation checks on every endpoint (not just authentication), JWTs signed with asymmetric keys so clients can't forge them, database users with minimum required permissions per service.

# Verify JWT signature with public key — prevents forgery
def verify_token(token: str) -> dict:
    try:
        payload = jwt.decode(
            token,
            PUBLIC_KEY,           # asymmetric — only the server can sign
            algorithms=['RS256'],  # reject HS256 to prevent "none" attack
            options={'verify_exp': True}
        )
        return payload
    except jwt.InvalidTokenError as e:
        raise AuthenticationError(f"Invalid token: {e}")

Running STRIDE in Practice

A useful format for a threat modelling session:

  1. Draw the data flow diagram (DFD) — 20 minutes
  2. For each process, data store, and data flow, apply all six STRIDE categories — 60 minutes
  3. Rate each threat by likelihood and impact — 20 minutes
  4. Assign mitigations and owners — 20 minutes
Two-hour investment: A two-hour STRIDE session for a new feature typically surfaces 8–15 threats, 3–5 of which are high severity. Fixing them during design costs a fraction of fixing them post-deployment — and far less than a security incident.

Threat modelling doesn't require a security specialist. Any developer who understands the system can run a productive STRIDE session. The value comes not from having the perfect answer but from asking the questions before writing the code.
