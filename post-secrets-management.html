Secrets Management Done Right: Vault, SOPS, and Why .env Files Are a Liability

Hardcoded secrets leak. A deep dive into HashiCorp Vault dynamic secrets, Mozilla SOPS for GitOps workflows, and a migration path from the .env file anti-pattern.

AR

Alex Rivera

Feb 28, 2024 10 min read

GitHub's secret scanning alerts fired over 1 million times in 2023. Developers accidentally committed API keys, database passwords, and private certificates to public repositories — and often to private ones that later became public, or were accessed by a compromised account. The root cause is almost always the same: secrets lived in files that were too easy to commit.

Why .env Files Are a Liability

The .env file pattern became popular because it works — it's simple and keeps secrets out of source code. The problems begin when:

  • Someone forgets to add .env to .gitignore and commits it
  • The file is copied to a shared drive or Slack message for "convenience"
  • You can't audit who has accessed which secret
  • You can't rotate secrets without updating files across multiple machines
  • Every developer has production credentials on their laptop
.env files externalise secrets from code but don't actually manage them. You've moved the problem, not solved it.

Option 1: HashiCorp Vault — Dynamic Secrets

Vault is the gold standard for secrets management. Its killer feature isn't just centralised storage — it's dynamic secrets. Instead of storing a long-lived database password, Vault generates a unique, time-limited credential for each service on demand, and revokes it automatically.

# Enable the database secrets engine
vault secrets enable database

# Configure the PostgreSQL connection
vault write database/config/myapp-db \
    plugin_name=postgresql-database-plugin \
    allowed_roles="app-role" \
    connection_url="postgresql://vault:{{password}}@postgres:5432/myapp" \
    username="vault" \
    password="vault-root-password"

# Create a role that generates credentials
vault write database/roles/app-role \
    db_name=myapp-db \
    creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN ENCRYPTED PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \
    default_ttl="1h" \
    max_ttl="24h"

Now each application instance gets a unique database credential that expires in one hour. If the application is compromised, the attacker's window is bounded. If you need to "rotate" credentials, you just stop issuing new ones — all existing credentials expire naturally.

# Application reads a fresh credential at startup (Python example)
import hvac

client = hvac.Client(url='https://vault.internal', token=os.environ['VAULT_TOKEN'])
db_creds = client.secrets.database.generate_credentials(name='app-role')

DB_USER = db_creds['data']['username']
DB_PASS = db_creds['data']['password']
Vault in Kubernetes: Use the Vault Agent Injector sidecar — it authenticates via Kubernetes service account tokens and writes secrets to a shared volume as files, so your application reads them without any Vault SDK code.

Option 2: Mozilla SOPS — Encrypted Secrets in Git

Not every team can deploy Vault. SOPS (Secrets OPerationS) offers a lighter-weight approach: secrets files are encrypted and committed to git. The encryption is transparent — you edit a YAML file, SOPS encrypts the values (not the keys) using AWS KMS, GCP KMS, Azure Key Vault, or age keys, and the ciphertext is safe to commit.

# Create an age key for local development
age-keygen -o ~/.config/sops/age/keys.txt

# Create a .sops.yaml configuration
# This tells SOPS which key to use for files in each path
creation_rules:
  - path_regex: environments/production/.*\.yaml$
    kms: arn:aws:kms:us-east-1:123456789:key/your-key-id
  - path_regex: environments/staging/.*\.yaml$
    kms: arn:aws:kms:us-east-1:123456789:key/staging-key-id
  - path_regex: .*\.yaml$
    age: age1xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
# Encrypt a secrets file
sops --encrypt secrets.yaml > secrets.enc.yaml

# Edit secrets in place (decrypts in memory, re-encrypts on save)
sops secrets.enc.yaml

# What gets committed to git — values are encrypted, keys are visible
database:
    password: ENC[AES256_GCM,data:7H3k...,type:str]
    host: ENC[AES256_GCM,data:aB2x...,type:str]

Option 3: Cloud-Native Secret Managers

If you're already in AWS, GCP, or Azure, their native secret managers (AWS Secrets Manager, GCP Secret Manager, Azure Key Vault) are excellent choices. They integrate naturally with IAM, support versioning and rotation, and have managed audit logs.

# Reading a secret from AWS Secrets Manager (Python)
import boto3
import json

client = boto3.client('secretsmanager', region_name='us-east-1')
response = client.get_secret_value(SecretId='prod/myapp/database')
secret = json.loads(response['SecretString'])
DB_PASS = secret['password']

The Migration Path

Moving away from .env files doesn't have to be a big-bang migration. A practical sequence:

  1. Audit first. Run git log --all --full-history -- "**/.env" to find every .env file ever committed. Rotate any credentials found — assume they're compromised.
  2. Add pre-commit hooks to prevent accidental commits: detect-secrets or gitleaks will scan for credential patterns before any commit goes through.
  3. Start new services using your chosen secret manager from day one.
  4. Migrate existing services one secret at a time during maintenance windows.
# .pre-commit-config.yaml — prevent secret commits
repos:
  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.4.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']
Developer experience matters: The reason .env files persist is that they're convenient. Whatever replaces them needs to be equally easy to use locally. Tools like direnv integrated with your secret manager can inject secrets into the shell environment transparently, giving developers the same ergonomics without the risk.

There is no scenario where a static secret in a file is the right long-term answer for production systems. The tools to do this properly — Vault, SOPS, cloud secret managers — are mature, well-documented, and far less painful to adopt than dealing with a credential compromise incident.
