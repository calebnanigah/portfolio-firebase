Performance Testing Under Load: k6 Patterns for Real-World Traffic

Simulating real user behaviour requires more than ramp-up scripts. This walkthrough covers soak tests, spike scenarios, and how to interpret results without drowning in metrics.

AR

Alex Rivera

Jan 30, 2024 13 min read

Most performance tests are too optimistic. They hammer a single endpoint with linear ramp-up, look at average response time, and call it done. Real traffic is bursty, multi-endpoint, and stateful — users log in, browse, add items to a cart, and checkout. A realistic performance test models that full journey. k6 is the tool that makes this approachable without sacrificing precision.

Why k6?

k6 scripts are written in JavaScript, which means they feel familiar to most development teams. But unlike browser-based tools, k6 runs at the protocol level — it generates HTTP requests without rendering a DOM, allowing a single machine to simulate thousands of concurrent virtual users. Tests live in version control, integrate naturally into CI pipelines, and emit structured metrics that plug directly into Grafana, Datadog, or the k6 Cloud dashboard.

The Four Load Test Patterns

1. Load Test — Establish the Baseline

A load test ramps up to your expected peak traffic and holds it for long enough to observe steady-state behaviour. The output tells you whether your system meets its SLA under normal peak load.

// load-test.js
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '2m', target: 100 },  // ramp up to 100 VUs
    { duration: '5m', target: 100 },  // hold at peak
    { duration: '1m', target: 0 },    // ramp down
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],  // 95th percentile under 500ms
    http_req_failed: ['rate<0.01'],    // error rate under 1%
  },
};

export default function () {
  const res = http.get('https://api.example.com/products');
  check(res, {
    'status is 200': (r) => r.status === 200,
    'response time OK': (r) => r.timings.duration < 500,
  });
  sleep(1);
}

2. Stress Test — Find the Breaking Point

A stress test pushes beyond expected load to find the point where the system degrades. You're looking for the VU count where error rates climb and response times blow out. This reveals your actual capacity ceiling, not a theoretical one.

export const options = {
  stages: [
    { duration: '2m', target: 100 },
    { duration: '2m', target: 200 },
    { duration: '2m', target: 400 },
    { duration: '2m', target: 800 },  // keep pushing
    { duration: '5m', target: 0 },    // let the system recover
  ],
};

3. Soak Test — Reveal Memory Leaks and Slow Degradation

A soak test runs at moderate load for an extended period — typically 4–8 hours. It catches resource leaks that don't appear in short-duration tests: memory that creeps upward, database connection pools that exhaust over time, or log files that fill a disk.

export const options = {
  stages: [
    { duration: '5m', target: 50 },   // ramp up
    { duration: '4h', target: 50 },   // hold — watch memory, CPU, GC pressure
    { duration: '5m', target: 0 },
  ],
};
Soak test insight: If p95 response time is 200ms at minute 5 and 800ms at hour 3 under the same load, you have a degradation problem. The most common culprits are connection pool exhaustion and memory leaks causing increased GC pauses.

4. Spike Test — Simulate Traffic Bursts

Traffic spikes happen — a product launch, a viral social post, a scheduled email campaign. Spike tests verify your autoscaling responds fast enough and that your system handles sudden load without data corruption or cascading failures.

export const options = {
  stages: [
    { duration: '1m', target: 10 },   // baseline
    { duration: '30s', target: 500 }, // sudden spike
    { duration: '2m', target: 500 },  // hold spike
    { duration: '30s', target: 10 },  // drop back
    { duration: '2m', target: 10 },   // observe recovery
  ],
};

Modelling Realistic User Journeys

Real users don't just hit one endpoint. Model the full journey using k6's scenario system to run multiple user flows concurrently in realistic proportions.

import { group } from 'k6';

export const options = {
  scenarios: {
    browse_products: {
      executor: 'ramping-vus',
      startVUs: 0,
      stages: [{ duration: '5m', target: 80 }],
      exec: 'browseFlow',
    },
    checkout: {
      executor: 'ramping-vus',
      startVUs: 0,
      stages: [{ duration: '5m', target: 20 }],
      exec: 'checkoutFlow',
    },
  },
};

export function browseFlow() {
  group('Browse', () => {
    http.get('/api/products?page=1');
    sleep(2);
    http.get('/api/products/123');
    sleep(1);
  });
}

export function checkoutFlow() {
  group('Checkout', () => {
    const cart = http.post('/api/cart', JSON.stringify({ productId: 123, qty: 1 }));
    check(cart, { 'cart updated': (r) => r.status === 200 });
    sleep(3);
    const order = http.post('/api/orders', JSON.stringify({ cartId: JSON.parse(cart.body).cartId }));
    check(order, { 'order placed': (r) => r.status === 201 });
  });
}

Interpreting Results Without Drowning

k6 emits dozens of metrics. Focus on these five to get 90% of the value:

  • http_req_duration p(95) and p(99) — percentiles reveal tail latency that averages hide
  • http_req_failed rate — anything above your SLA threshold is a problem
  • http_reqs rate — actual throughput achieved, not just what you targeted
  • vus_max — confirm the test reached your target concurrency
  • data_received / data_sent — bandwidth bottlenecks show up here
Pipeline integration: Run load tests in a dedicated CI stage triggered by merges to main, not on every PR. Store results as artifacts and trend p95 over time. A performance regression is a performance regression even if no threshold was breached today.

Performance testing is most valuable when it's continuous, not a pre-launch ritual. Build it into your delivery pipeline, set thresholds that reflect real SLAs, and treat a performance regression the same way you treat a functional bug — as a defect that must be fixed before shipping.
