Machine Learning for Well Log Interpretation: A Practical Python Guide

Lithology prediction from LAS files using gradient boosting and neural networks. Covers data cleaning, feature engineering from GR/RHOB/NPHI logs, and model evaluation with real field data.

AR

Alex Rivera

Mar 10, 2024 18 min read

Petrophysicists have been interpreting well logs manually for decades — a skilled analyst can look at a GR/RHOB/NPHI/RT crossplot and identify sand, shale, carbonate, and fluid contacts with impressive accuracy. The challenge is scale: a single horizontal well in a tight formation may generate thousands of depth intervals to interpret, and a field development programme might drill dozens of wells in parallel. Machine learning doesn't replace petrophysical expertise — it multiplies it.

The Data: LAS Files and the Volve Dataset

Well log data is stored in LAS (Log ASCII Standard) format. The lasio library makes loading these files in Python straightforward. We'll use the Equinor Volve dataset, a fully open North Sea field dataset that includes 24 wells with comprehensive log suites.

import lasio
import pandas as pd
import numpy as np

def load_well_logs(las_path: str) -> pd.DataFrame:
    """Load a LAS file and return a tidy DataFrame."""
    las = lasio.read(las_path)
    df = las.df().reset_index()
    df.columns = [col.upper() for col in df.columns]

    # Standard log mnemonics we'll use
    required_logs = ['DEPTH', 'GR', 'RHOB', 'NPHI', 'RT', 'DT']
    available = [col for col in required_logs if col in df.columns]
    df = df[available].copy()

    # Replace null values (-999.25 is the LAS convention)
    df.replace(-999.25, np.nan, inplace=True)

    return df

# Load all wells
import glob
wells = {}
for las_file in glob.glob('volve_data/*.las'):
    well_name = las_file.split('/')[-1].replace('.las', '')
    wells[well_name] = load_well_logs(las_file)

print(f"Loaded {len(wells)} wells")

Data Cleaning and Quality Control

Raw well logs are messy: washout zones cause density and neutron readings to be unreliable, tool failures create spikes, and depth mismatches occur when multiple logging runs are merged. QC is the most time-consuming part of any ML workflow on petrophysical data.

def qc_logs(df: pd.DataFrame) -> pd.DataFrame:
    """Apply basic QC filters to well log data."""
    df = df.copy()

    # Flag unrealistic density values (bulk density of rock: 1.8–3.0 g/cc)
    if 'RHOB' in df.columns:
        df.loc[(df['RHOB'] < 1.8) | (df['RHOB'] > 3.0), 'RHOB'] = np.nan

    # Flag unrealistic neutron porosity (-0.1 to 0.6 in fractional units)
    if 'NPHI' in df.columns:
        df.loc[(df['NPHI'] < -0.1) | (df['NPHI'] > 0.6), 'NPHI'] = np.nan

    # Remove intervals where more than 2 logs are missing
    log_cols = [c for c in ['GR', 'RHOB', 'NPHI', 'RT', 'DT'] if c in df.columns]
    df['null_count'] = df[log_cols].isnull().sum(axis=1)
    df = df[df['null_count'] <= 2].copy()
    df.drop(columns=['null_count'], inplace=True)

    return df

Feature Engineering

Raw log values are informative, but derived features often carry more signal for lithology classification. The neutron-density crossplot is a classic tool: the separation between NPHI and RHOB characterises gas-bearing vs. brine-saturated sands, and both contrast with carbonate and shale signatures.

def engineer_features(df: pd.DataFrame) -> pd.DataFrame:
    """Create derived petrophysical features for ML."""
    df = df.copy()

    # Neutron-Density separation (gas indicator in sands)
    if 'NPHI' in df.columns and 'RHOB' in df.columns:
        df['NDSEP'] = df['NPHI'] - (2.65 - df['RHOB']) / (2.65 - 1.0)

    # Normalised GR (0=clean sand, 1=pure shale) using Larionov correction
    if 'GR' in df.columns:
        gr_min, gr_max = df['GR'].quantile(0.05), df['GR'].quantile(0.95)
        igr = (df['GR'] - gr_min) / (gr_max - gr_min)
        igr = igr.clip(0, 1)
        # Larionov correction for older rocks
        df['VSH'] = 0.33 * (2 ** (2 * igr) - 1)

    # Resistivity log ratio (hydrocarbon indicator)
    if 'RT' in df.columns:
        df['RT_LOG'] = np.log10(df['RT'].clip(0.1, 10000))

    # Depth-normalised features (removes regional trend)
    for col in ['GR', 'RHOB', 'NPHI']:
        if col in df.columns:
            rolling_mean = df[col].rolling(window=50, center=True, min_periods=10).mean()
            df[f'{col}_NORM'] = df[col] - rolling_mean

    return df

Training a Lithology Classifier

With labelled intervals from cored wells (where rock has been physically retrieved and described), we can train a supervised classifier. LightGBM handles the mixed numerical/missing data well and trains quickly enough for iterative experimentation.

from sklearn.model_selection import GroupShuffleSplit
from sklearn.preprocessing import LabelEncoder
from sklearn.metrics import classification_report
import lightgbm as lgb
import pandas as pd

# Load labelled core data (manually described lithologies)
core_df = pd.read_csv('volve_core_descriptions.csv')
# Columns: WELL, DEPTH_MD, LITHOLOGY (Sand, Shale, Carbonate, Coal)

# Join core labels to log features
feature_cols = ['GR', 'RHOB', 'NPHI', 'RT_LOG', 'VSH', 'NDSEP',
                'GR_NORM', 'RHOB_NORM', 'NPHI_NORM']

# Group split by well — prevents data leakage between wells
gss = GroupShuffleSplit(n_splits=1, test_size=0.25, random_state=42)
train_idx, test_idx = next(gss.split(core_df, groups=core_df['WELL']))

X_train = core_df.iloc[train_idx][feature_cols]
X_test = core_df.iloc[test_idx][feature_cols]

le = LabelEncoder()
y_train = le.fit_transform(core_df.iloc[train_idx]['LITHOLOGY'])
y_test = le.transform(core_df.iloc[test_idx]['LITHOLOGY'])

model = lgb.LGBMClassifier(
    n_estimators=500,
    learning_rate=0.05,
    num_leaves=63,
    min_child_samples=20,
    class_weight='balanced',
    random_state=42
)
model.fit(X_train, y_train,
          eval_set=[(X_test, y_test)],
          callbacks=[lgb.early_stopping(50, verbose=False)])

y_pred = model.predict(X_test)
print(classification_report(y_test, y_pred, target_names=le.classes_))
Critical: split by well, not by depth. Splitting the dataset randomly by depth interval causes data leakage — adjacent samples are nearly identical, so the model appears to perform perfectly but fails completely on unseen wells. Always use a group split where groups are wells.

Applying the Model to Uncored Wells

Once trained and validated, the model can predict lithology continuously across any well with the required log suite, including wells without core.

def predict_lithology(model, le, well_df: pd.DataFrame,
                       feature_cols: list) -> pd.DataFrame:
    """Apply trained model to a well's log data."""
    well_df = well_df.copy()
    X = well_df[feature_cols].copy()

    # Handle missing features gracefully
    for col in feature_cols:
        if col not in X.columns:
            X[col] = np.nan

    # Predict only where we have enough data
    valid_mask = X.isnull().sum(axis=1) <= 2
    well_df['LITHO_PRED'] = np.nan
    well_df['LITHO_CONF'] = np.nan

    proba = model.predict_proba(X[valid_mask])
    well_df.loc[valid_mask, 'LITHO_PRED'] = le.inverse_transform(proba.argmax(axis=1))
    well_df.loc[valid_mask, 'LITHO_CONF'] = proba.max(axis=1)

    return well_df
Model interpretability: LightGBM's feature importance plots reveal which logs drive predictions. GR typically dominates for shale/sand discrimination, while NPHI-RHOB crossplot features are most important for carbonate vs. sand. This matches classical petrophysical knowledge — a good sign the model has learned physics, not just correlations.

Going Further: Sequential Models for Vertical Continuity

The tree-based model treats each depth sample independently, ignoring the vertical structure of stratigraphy. Lithologies form layers — a LSTM or 1D CNN that processes the log sequence as a time series can leverage this structure. For a 200m interval with 0.1m sampling, you have a 2000-step sequence — well within LSTM range.

The combination of domain expertise (feature engineering, QC, petrophysical rules) with modern gradient boosting or deep learning consistently outperforms either approach alone. The domain knowledge guides the model towards physically meaningful representations; the model handles the scale and consistency that human analysts cannot match.
