API Contract Testing with Pact: Stop Breaking Microservices

Consumer-driven contract testing lets teams evolve microservices independently without regression surprises. A hands-on guide using Pact in a Node.js/TypeScript stack.

AR

Alex Rivera

Mar 12, 2024 11 min read

In a microservices architecture, each service is independently deployable — in theory. In practice, teams are terrified to ship because nobody knows if their changes will silently break a downstream consumer. Integration test suites that spin up multiple services help, but they're slow, brittle, and expensive to maintain. Contract testing is the answer: instead of testing the integration live, you test that each side honours a shared contract.

What Is Contract Testing?

Contract testing verifies that a consumer (a service that calls an API) and a provider (the service that serves the API) agree on the shape of the interaction. Pact is the most widely adopted framework for this pattern. The consumer writes tests that describe what it sends and what it expects to receive. Pact records these interactions as a pact file (a JSON contract). The provider then verifies that it can fulfil every interaction described in that file.

Contract tests give you the confidence of an integration test at the speed of a unit test. Each side runs independently without requiring the other to be alive.

The critical shift is that the contract is owned by the consumer. This prevents providers from making breaking changes without knowing who depends on them.

Setting Up Pact in a TypeScript Consumer

Install the Pact JS library and a test runner. We'll use Jest here.

npm install --save-dev @pact-foundation/pact jest ts-jest

Define a consumer test for a UserService that calls a downstream AccountService:

// user-service/src/__tests__/accountService.pact.test.ts
import { Pact, Matchers } from '@pact-foundation/pact';
import path from 'path';
import { fetchAccount } from '../clients/accountClient';

const { like, term } = Matchers;

const provider = new Pact({
  consumer: 'UserService',
  provider: 'AccountService',
  port: 4000,
  log: path.resolve(process.cwd(), 'logs', 'pact.log'),
  dir: path.resolve(process.cwd(), 'pacts'),
  logLevel: 'warn',
});

describe('AccountService Client', () => {
  beforeAll(() => provider.setup());
  afterAll(() => provider.finalize());
  afterEach(() => provider.verify());

  describe('GET /accounts/:id', () => {
    beforeEach(() =>
      provider.addInteraction({
        state: 'account 42 exists',
        uponReceiving: 'a request for account 42',
        withRequest: {
          method: 'GET',
          path: '/accounts/42',
          headers: { Accept: 'application/json' },
        },
        willRespondWith: {
          status: 200,
          headers: { 'Content-Type': 'application/json' },
          body: {
            id: like(42),
            email: term({ generate: 'user@example.com', matcher: '^[^@]+@[^@]+\\.[^@]+$' }),
            plan: like('pro'),
          },
        },
      })
    );

    it('returns account data for a valid ID', async () => {
      const account = await fetchAccount(42);
      expect(account.id).toBe(42);
      expect(account.plan).toBeDefined();
    });
  });
});
Note on matchers: Use like() for type-matching (any string, any number) rather than exact values. This makes contracts more resilient to data changes while still enforcing shape.

Publishing and Verifying with Pact Broker

Raw pact files on disk don't scale. Pact Broker is a hosted service (or self-hosted via Docker) that stores contracts and provides webhooks to trigger provider verification on every consumer contract change.

# docker-compose.yml for self-hosted Pact Broker
version: '3'
services:
  pact-broker:
    image: pactfoundation/pact-broker:latest
    ports:
      - "9292:9292"
    environment:
      PACT_BROKER_DATABASE_URL: "sqlite:////tmp/pact_broker.sqlite3"
      PACT_BROKER_BASE_URL: "http://localhost:9292"

Publish pacts from your CI pipeline after the consumer tests run:

# In your CI pipeline (GitHub Actions example)
- name: Publish pacts
  run: npx pact-broker publish ./pacts \
    --broker-base-url $PACT_BROKER_URL \
    --consumer-app-version $GITHUB_SHA \
    --branch $GITHUB_REF_NAME

Provider Verification

On the provider side, Pact replays every recorded interaction against the running service and checks the responses. Provider states (like "account 42 exists") are set up using state handlers — typically fixtures or database seeds.

// account-service/src/__tests__/pact.provider.test.ts
import { Verifier } from '@pact-foundation/pact';
import { startServer } from '../server';

describe('Provider verification', () => {
  let server: ReturnType<typeof startServer>;

  beforeAll(async () => {
    server = await startServer(4001);
  });

  afterAll(() => server.close());

  it('validates the provider against its consumers', async () => {
    await new Verifier({
      provider: 'AccountService',
      providerBaseUrl: 'http://localhost:4001',
      pactBrokerUrl: process.env.PACT_BROKER_URL,
      publishVerificationResult: true,
      providerVersion: process.env.GITHUB_SHA,
      stateHandlers: {
        'account 42 exists': async () => {
          await db.seed({ id: 42, email: 'user@example.com', plan: 'pro' });
        },
        'no accounts exist': async () => {
          await db.clear();
        },
      },
    }).verifyProvider();
  });
});

Can I Deploy? — The Safety Gate

The Pact Broker exposes a can-i-deploy command. Before deploying either service, your CI pipeline asks the broker: "Is this version of UserService compatible with the currently deployed version of AccountService?" If any pact verification has failed, the deploy is blocked.

npx pact-broker can-i-deploy \
  --pacticipant UserService \
  --version $GITHUB_SHA \
  --to-environment production \
  --broker-base-url $PACT_BROKER_URL
The payoff: Teams using contract testing routinely report eliminating 80–90% of their microservice integration incidents. More importantly, they ship faster because the fear of breaking something downstream is replaced by a concrete, automated answer.

Common Pitfalls

  • Testing too much in the contract. Contracts should cover the minimum surface the consumer actually uses, not the entire provider API. Over-specified contracts break unnecessarily.
  • Skipping state handlers. Without provider states, verification runs against whatever data happens to be in the database, leading to flaky results.
  • Not running verification in CI. A pact only provides safety when verification is automated and tied to the deployment pipeline.

Contract testing isn't a replacement for all integration testing, but for the specific problem of microservice API compatibility, it is vastly more effective than any alternative. If your team is deploying services independently and relying on hope as a strategy, Pact is the tool that turns hope into certainty.
