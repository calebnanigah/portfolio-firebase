Writing Flake-Free E2E Tests: Lessons from 50,000 Playwright Runs

Flaky tests erode trust faster than no tests. After running Playwright at scale, here are the patterns that eliminate 95% of non-deterministic failures.

AR

Alex Rivera

Feb 19, 2024 7 min read

A flaky test is worse than no test. It trains developers to ignore failures, undermines confidence in the entire suite, and wastes hours investigating non-issues. After analysing failure patterns across tens of thousands of Playwright runs, a handful of root causes account for the overwhelming majority of flakiness. Most of them are entirely preventable.

The Anatomy of a Flaky Test

Flakiness is almost always a timing or isolation problem. Either the test is interacting with the application before it's ready, or the state left by one test is leaking into another. The solution to both is precision — tell Playwright exactly what to wait for, and ensure each test starts from a known clean state.

If you've ever written await page.waitForTimeout(2000), that's a flakiness time-bomb. Fixed delays are guesses dressed as code.

Rule 1: Never Use Fixed Timeouts

Fixed delays (waitForTimeout) are the single biggest source of flakiness. They're too short on a slow CI machine and wastefully long on fast hardware. Replace every fixed delay with an actionable wait.

// Bad — arbitrary sleep
await page.waitForTimeout(2000);
await page.click('#submit-btn');

// Good — wait for the element to be ready
await page.waitForSelector('#submit-btn', { state: 'visible' });
await page.click('#submit-btn');

// Better — Playwright's locators auto-wait
await page.getByRole('button', { name: 'Submit' }).click();

Playwright's locator API automatically waits for elements to be actionable before interacting with them. Use locators everywhere.

Rule 2: Wait for Network and Application State

Clicking a button that triggers an API call and then immediately asserting on the result is a race condition. Wait for the network response, or better yet, wait for the UI change that proves the data loaded.

// Wait for a specific API response
const responsePromise = page.waitForResponse(
  resp => resp.url().includes('/api/orders') && resp.status() === 200
);
await page.getByRole('button', { name: 'Load Orders' }).click();
await responsePromise;

// Then assert on the rendered result
await expect(page.getByTestId('order-list')).toBeVisible();
await expect(page.getByTestId('order-item')).toHaveCount(5);

Rule 3: Isolate Test State

Tests that share state are tests waiting to fail. Each test should set up its own data and clean up after itself, or run in a completely isolated environment (separate database, separate browser context).

// playwright.config.ts — use browser contexts for isolation
import { defineConfig } from '@playwright/test';

export default defineConfig({
  use: {
    // Each test gets a fresh browser context
    contextOptions: { storageState: undefined },
  },
});

// In test file — create fresh data per test
test.beforeEach(async ({ request }) => {
  // Seed test data via API rather than through the UI
  await request.post('/api/test/seed', {
    data: { user: { email: `test-${Date.now()}@example.com` } }
  });
});

Rule 4: Use data-testid Attributes — Sparingly

CSS selectors tied to visual styling break when the design changes. XPath is fragile. The most resilient selectors target semantic roles and data-testid attributes that survive refactoring.

// Fragile — breaks on CSS class rename
await page.click('.btn.btn-primary.submit-action');

// Resilient — semantic role selector
await page.getByRole('button', { name: 'Place Order' }).click();

// Explicit test hook for complex components
await page.getByTestId('checkout-confirm-btn').click();
Accessibility bonus: Writing role-based selectors forces you to use proper semantic HTML and ARIA labels. Your tests improve accessibility as a side effect.

Rule 5: Run Tests in Parallel with Shared Fixtures

Parallelism exposes implicit state dependencies that sequential runs hide. Enable Playwright's worker-level parallelism and use fixtures to share expensive setup (like authenticated sessions) without sharing mutable state.

// fixtures/auth.ts
import { test as base } from '@playwright/test';

export const test = base.extend<{ authenticatedPage: Page }>({
  authenticatedPage: async ({ page }, use) => {
    // Log in once per worker, share session via storage state
    await page.goto('/login');
    await page.fill('[name=email]', process.env.TEST_USER_EMAIL!);
    await page.fill('[name=password]', process.env.TEST_USER_PASSWORD!);
    await page.click('[type=submit]');
    await page.waitForURL('/dashboard');
    await use(page);
  },
});

Rule 6: Treat Flakes as Bugs

When a test flakes, quarantine it immediately (mark it with test.skip or move to a dedicated flaky suite), investigate the root cause, and only re-enable it once fixed. Never allow --retries=3 to become a substitute for fixing the underlying problem — retries are a reliability band-aid for infrastructure issues, not a licence to write non-deterministic tests.

Red flag: If you're relying on retries: 3 for more than 2% of your test suite, you have a flakiness problem that retries are masking. Fix the tests.

The Flake Dashboard

Track flakiness metrics over time. Log test names, failure messages, and timestamps. A test that fails 1 in 20 runs is costing your team far more than the time to fix it. Prioritise by frequency multiplied by disruption cost — the tests that run most often and fail most publicly are your highest-value targets.

Flake-free Playwright suites are achievable. The key insight is that almost all flakiness is deterministic in origin — there's a real race condition, a real state leak, a real selector that's too fragile. Fix the test, fix the root cause, and build a suite the team actually trusts.
