OWASP Top 10 in 2024: What's Changed and What Still Gets You Hacked

Injection is no longer #1. Broken access control now tops the list. A practical breakdown of each category with code-level examples developers can act on today.

AR

Alex Rivera

Apr 2, 2024 14 min read

The OWASP Top 10 is the de facto standard reference for web application security. Updated every few years, it reflects real-world vulnerability data gathered from hundreds of organisations and thousands of applications. The 2021 edition — still current as of 2024 — made some dramatic shifts that every developer should understand.

What Changed: The Big Three Movements

A01 — Broken Access Control jumped from fifth to first. It's the most prevalent vulnerability in tested applications. Injection dropped to third. Three new categories appeared: Insecure Design, Software and Data Integrity Failures, and Server-Side Request Forgery (SSRF).

The shift from Injection to Broken Access Control at the top tells us something important: frameworks have largely solved injection at the library level, but authorisation logic is still hand-rolled and still wrong.

A01: Broken Access Control

Access control enforces that users can only do what they're permitted to do. When it breaks, authenticated users escalate privileges, access other users' data, or perform actions they shouldn't. The most common failure is trusting the client to supply their own user ID.

# Vulnerable — user supplies their own ID in the request body
# POST /api/account/update
# {"userId": 999, "email": "attacker@evil.com"}

@app.route('/api/account/update', methods=['POST'])
def update_account():
    data = request.json
    user_id = data['userId']  # NEVER trust this
    db.execute("UPDATE users SET email=? WHERE id=?",
               (data['email'], user_id))

# Secure — derive userId from the authenticated session token
@app.route('/api/account/update', methods=['POST'])
def update_account():
    user_id = get_current_user_id()  # from JWT/session, not from request
    data = request.json
    db.execute("UPDATE users SET email=? WHERE id=?",
               (data['email'], user_id))

A02: Cryptographic Failures

Formerly called "Sensitive Data Exposure," this category covers failures in how data is protected in transit and at rest. The most common failures: storing passwords with MD5 or SHA1 instead of bcrypt/Argon2, transmitting sensitive data over HTTP, and using weak or hardcoded encryption keys.

# Dangerous — MD5 is not a password hashing algorithm
import hashlib
password_hash = hashlib.md5(password.encode()).hexdigest()

# Correct — use bcrypt with appropriate cost factor
import bcrypt
password_hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))

A03: Injection

SQL injection, command injection, LDAP injection — any time user input is interpreted as code rather than data. Modern ORMs largely prevent SQL injection, but developers still reach for raw query strings when they think they need performance or flexibility.

# Vulnerable — string interpolation in a SQL query
query = f"SELECT * FROM orders WHERE user_id = {user_id} AND status = '{status}'"

# Secure — parameterised query
cursor.execute(
    "SELECT * FROM orders WHERE user_id = %s AND status = %s",
    (user_id, status)
)

A04: Insecure Design

New in 2021, this category acknowledges that some vulnerabilities are architectural, not implementation bugs. Insufficient rate limiting, missing multi-factor authentication on sensitive operations, password reset flows that don't expire tokens — these are design failures that no amount of code review catches because the right code was never specified.

Threat modelling helps here. Running a STRIDE analysis during design surfaces "what if an attacker does X" questions before any code is written. See the companion post on STRIDE for a hands-on walkthrough.

A05: Security Misconfiguration

Default credentials, verbose error messages exposing stack traces, unnecessary open ports, permissive CORS policies, missing security headers. This is the "everything else" category and it's genuinely everywhere.

# Check for missing security headers with a simple curl
curl -I https://yourapp.com | grep -E "X-Frame-Options|X-Content-Type|CSP|HSTS"

# Expected headers to set in your web framework:
# Strict-Transport-Security: max-age=31536000; includeSubDomains
# X-Content-Type-Options: nosniff
# X-Frame-Options: DENY
# Content-Security-Policy: default-src 'self'

A06: Vulnerable and Outdated Components

Log4Shell, the Spring4Shell RCE, the XZ Utils backdoor — the most devastating attacks in recent years targeted third-party libraries, not custom application code. Knowing what's in your supply chain is now a security requirement, not a nice-to-have.

# Automate dependency scanning in CI
# npm
npm audit --audit-level=high

# Python
pip-audit

# Docker images
trivy image myapp:latest --exit-code 1 --severity HIGH,CRITICAL

A07: Identification and Authentication Failures

Weak password policies, missing account lockout, broken "remember me" tokens, session IDs in URLs. Authentication is a solved problem — use a proven identity library or managed auth provider rather than building it yourself.

A08: Software and Data Integrity Failures

New in 2021, this covers CI/CD pipeline compromises and insecure deserialization. The SolarWinds attack was a textbook A08 failure — malicious code was injected into a legitimate software build. Verify the integrity of your build pipeline and sign your artefacts.

A09: Security Logging and Monitoring Failures

You can't detect an incident if you're not logging security events. Failed logins, privilege escalations, unusual data access patterns — all must be logged with enough context to reconstruct what happened, and those logs must be monitored for anomalies.

A10: Server-Side Request Forgery (SSRF)

New in 2021 and increasingly important as cloud adoption grows. SSRF occurs when an application fetches a remote resource based on user-supplied input without validating the destination URL. An attacker can use this to probe internal services, access cloud metadata endpoints (169.254.169.254), or exfiltrate data.

# Vulnerable
import requests
url = request.args.get('url')
response = requests.get(url)  # attacker supplies http://169.254.169.254/latest/meta-data/

# Secure — allowlist permitted domains
from urllib.parse import urlparse
ALLOWED_HOSTS = {'api.trusted-partner.com', 'cdn.example.com'}

def safe_fetch(url):
    parsed = urlparse(url)
    if parsed.hostname not in ALLOWED_HOSTS:
        raise ValueError(f"Host {parsed.hostname} not permitted")
    return requests.get(url, timeout=5)

The OWASP Top 10 is a starting point, not a comprehensive security programme. But addressing all ten categories systematically puts you ahead of the majority of applications in production today. Start with broken access control — it's the most prevalent, and fixing it typically requires examining your authorisation model end-to-end, which surfaces issues in categories four through seven as a side effect.
